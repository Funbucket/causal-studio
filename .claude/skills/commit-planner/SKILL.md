# Commit Planner

코드 변경사항을 분석하여 논리적인 커밋 단위로 그룹화하고 커밋 계획을 작성한다.

## 워크플로우

### 1. 변경사항 수집

```bash
git status
git diff --stat
git diff HEAD
```

### 2. 변경사항 분석

각 파일의 변경 내용을 분석하여 다음을 파악한다:
- 변경 목적 (신규 기능, 버그 수정, 리팩토링 등)
- 관련 컴포넌트/모듈
- 의존 관계

### 3. 커밋 단위 그룹화

**원칙:**
- 하나의 커밋 = 하나의 명확한 목적
- 관련 파일들을 함께 묶음
- 의존성 순서 고려 (기반 → 활용)

**그룹화 기준:**
1. 공통 훅/유틸리티 생성
2. 공통 컴포넌트 생성
3. 페이지별 적용
4. 스타일/설정 변경
5. 버그 수정

### 4. 커밋 메시지 작성

**형식:**
```
<type>(scope): <한국어 설명>
```

**타입:**
| type | 용도 |
|------|------|
| feat | 새로운 기능 추가 |
| fix | 버그 수정 |
| refactor | 코드 리팩토링 |
| style | 스타일/포맷팅 변경 |
| chore | 기타 변경 (설정, 빌드 등) |
| docs | 문서 변경 |
| test | 테스트 추가/수정 |

**예시:**
```
feat(matching): matching estimator 분석용 ipynb 추가
fix(matching): 씬 하이라이트 박스와 수식 겹침 문제 수정
refactor(matching): 공통 애니메이션 로직 유틸 함수로 분리
```

## 출력 형식

```markdown
# 커밋 계획

## 커밋 1: <간략한 설명>

**메시지:**
```
<type>(scope): <한국어 설명>
```

**포함 파일:**
- `path/to/file1.py`
- `path/to/file2.md`

**변경 내용:**
- <변경 사항 1>
- <변경 사항 2>

---

## 커밋 2: ...
```

## 주의사항

- 각 커밋은 독립적으로 빌드/테스트 가능해야 함
- 의존성이 있는 변경은 의존 대상을 먼저 커밋
- 너무 큰 커밋은 분리, 너무 작은 커밋은 병합 고려
